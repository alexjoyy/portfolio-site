Case Study 02

Payments Reconciliation Service

A reconciliation engine that compares payment gateway records with internal ledger entries, classifies mismatches, and exposes actionable reports for operations teams.

Java 21 Spring Boot Scheduler PostgreSQL Render Deployment
GitHub Repository Live Health Endpoint

Architecture Decisions

  • Separated ingestion, reconciliation runner, and reporting concerns.
  • Tracked each run with run-level metrics for auditability.
  • Modeled unmatched records as first-class output for support workflows.

Key Challenge

Data could arrive at different times from gateway and ledger sources. Reconciliation logic was designed to handle asynchronous arrival without losing mismatch traceability.

Request Flow

  • `POST /api/reconciliation/gateway-transactions` ingests gateway records.
  • `POST /api/reconciliation/ledger-entries` ingests internal entries.
  • `POST /api/reconciliation/run` executes matching logic.
  • `GET /api/reconciliation/reports/unmatched` returns exception set.

Outcome

  • Fast visibility into financial mismatches and missing entries.
  • Repeatable reconciliation runs with clear counts and timestamps.
  • Ready foundation for CSV exports and scheduled operations monitoring.
